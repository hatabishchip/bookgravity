# МЕТАПРОМТ: Делегируемые права тренера + запись без кода при недоставке OTP

v1 - 06.07.2026. Запрос: голосовое Светланы (Ячинта ведёт WhatsApp-чат с лидами из
рекламы, но не может записывать клиентов к другим тренерам) + требование владельца
(staff никогда не вводит WhatsApp-код за клиента) + кейс клиента с китайским номером
(код WhatsApp не доставляется - клиент не может записаться сам).

## Цель

1. Админ студии может делегировать тренерам ЧАСТЬ админских функций - тумблерами
   на каждого тренера отдельно («на своё усмотрение»):
   - **Права записи**: видеть всё расписание студии и записывать клиентов в ЛЮБОЙ
     класс любого тренера на любую будущую дату.
   - **Права управления записями**: переносить запись клиента в другой класс и
     отменять записи клиентов.
   - Само расписание (создание/изменение/отмена классов) тренеру НЕ доступно.
2. Staff (админ и тренер) НИКОГДА не вводит WhatsApp-код при записи клиента -
   ни в кабинете, ни на публичной странице, ни в приложении (подтверждено кодом,
   закрепляется тестом).
3. Клиент, которому код WhatsApp не доставился (номер вне WhatsApp, Китай),
   может записаться без кода; запись помечается «номер не подтверждён».

## Факты из кода (проверено 06.07.2026)

- `/api/bookings` (публичная запись): `isStaffOfStudio` → staff пропускает OTP. ЕСТЬ.
- `/api/otp/send`: staff → `{skipped:true, staff:true}`, виджет не показывает код. ЕСТЬ.
- `POST /api/trainer/bookings`: без OTP, НО только СВОЙ класс (403) и только СЕГОДНЯ (403).
- `POST /api/admin/bookings`: без OTP. ЕСТЬ.
- Роли: ADMIN / TRAINER / STAFF / SUPER_ADMIN. Делегирования нет.
- Виджет при `delivery=failed` просит сменить номер - записаться нельзя.
- +86 (Китай) в таблице стран есть (11 цифр); блокер - недоставка кода, не валидация.

## Реализация

### A. Схема (Prisma + миграция Turso по паттерну репо)
На `Trainer` два поля:
- `permBookAnyClass  Boolean @default(false)` - расписание студии + запись в любой класс.
- `permManageBookings Boolean @default(false)` - перенос/отмена записей клиентов.
На `Booking`:
- `phoneUnverified Boolean @default(false)` - запись создана без подтверждения кода
  (код не доставился).
Миграция: `prisma/migrations/<ts>_trainer_delegation/migration.sql` (ADD COLUMN ×3)
+ скрипт `scripts/apply-trainer-delegation-migration.ts` (@libsql/client, идемпотентно).

### B. API
1. Админ, карточка тренера (`/api/admin/trainers/[id]`): принять и сохранять оба тумблера.
2. НОВОЕ `GET /api/trainer/studio-schedule` (гейт `permBookAnyClass`): предстоящие классы
   всей студии - дата, время, тренер, свободные места, цена.
3. `POST /api/trainer/bookings`: при `permBookAnyClass` разрешить любой слот студии
   (не только свой) и будущие даты (сегодня + вперёд; классы, которые уже закончились, -
   нельзя). Без права - поведение как сейчас (свой класс, сегодня). OTP - нет и не будет.
4. Перенос/отмена записи (гейт `permManageBookings`):
   - `PATCH /api/trainer/bookings/[id]` действие reschedule (в другой слот с местами),
   - отмена: `cancelledByRole="trainer"`, возврат membership-класса как в админ-отмене.
   Логика зеркалит существующие админ-эндпоинты (не дублируя: вынести общее в lib при
   необходимости).
5. `/api/bookings` (публичный): если OTP для телефона имеет `delivery=failed` (проверка
   на сервере по последнему коду этого телефона+студии), пропустить проверку кода и
   создать запись с `phoneUnverified=true`. IP-лимит остаётся.

### C. UI
1. Админка → Trainers: секция «Delegated rights» с двумя тумблерами (EN, как весь админ-UI).
2. Кабинет тренера: вкладка «Studio schedule» (видна только с правом) - список классов
   с местами и кнопкой добавить клиента (имя/телефон/email, тир цены); при
   `permManageBookings` у записей - перенести/отменить.
3. Виджет записи: при `delivery=failed` - вместо тупика кнопка продолжить без кода
   (текст EN, дефис, без тире); запись создаётся, помечается.
4. Бейдж «phone unverified» в списках записей админа и тренера.

### D. Уведомления
- Запись, созданная тренером в чужой класс, шлёт клиенту стандартное WhatsApp/email
  подтверждение и уведомляет ведущего тренера класса (существующий пайплайн).
- Отмена/перенос тренером - клиентские уведомления как при админ-действии.
- Клиентские ТЕКСТЫ не меняются (правило шаблонов); используются существующие.

### E. Проверка (обязательно перед «готово»)
1. `npx tsc --noEmit` + сборка.
2. Матрица прав: тренер без прав (старое поведение), с правом записи, с обоими;
   попытки обхода (чужой слот без права = 403, прошедший класс = 403).
3. OTP: staff-пути не спрашивают код (все три: админка, кабинет, публичная под логином);
   клиент с failed-доставкой записывается, бейдж виден.
4. Деплой на Vercel (git push, команда hatabishchips-projects), проверка live.
5. Память: обновить project_bookgravity + этот файл (статусы).

## Допущения (явно)
- Права per-trainer, per-studio (Trainer и так привязан к студии).
- «Любая дата» = сегодня и будущее; в прошедшие классы записывать нельзя.
- Staff-запись игнорирует публичный cutoff 2.5 ч (как сейчас в своём классе сегодня),
  но не позже конца класса.
- Тексты интерфейса кабинета/админки - EN (как весь staff-UI).
- Сообщение Светлане о новых возможностях готовит владелец (или я по отдельному
  одобрению) - не часть этой задачи.
