# Метапромт: тренеру - английский черновик, клиенту - его язык (Canggu)

Утверждён кнопками: охват = только Canggu (в Ubud агент не подключён; агент-студия = {canggu}).
Ubud не трогаем (остаётся inboxLanguage=ru для Светы, агента там нет).

## Цель (желаемое поведение, слова владельца 22.07)

Клиенту отвечаем на ТОМ языке, на котором он написал (английский -> английский, русский ->
русский, Bahasa -> Bahasa). А тренеру (Сеня, Canggu) черновик агента на подтверждение перед
отправкой ВСЕГДА показываем на английском. То есть: язык персонала (что видит и правит тренер)
= английский, язык доставки (что получает клиент) = язык клиента.

## Почему сейчас не так (найдено в коде 22.07)

1. `lib/sales-agent.ts` инструктирует агента «reply in the client's language» -> черновик на
   языке клиента, и в треде Вероники агент ушёл в русский, потому что в истории уже был русский
   исходящий (модель продолжила язык треда). Гарантии «тренеру всегда английский» нет.
2. Инбокс показывает черновик агента КАК ЕСТЬ (`detail.suggestion.draft`, Inbox.tsx ~2293), без
   перевода для показа. Значит язык, на котором агент написал = язык, который видит тренер.
3. `inboxLanguage` студии агент вообще не читает (в sales-agent.ts его нет).

## Решение (две части)

### A. Настройка студии Canggu: inboxLanguage = "en"
(Сегодня я по ошибке поставил "off" под неверную трактовку - вернуть на "en".)
Это даёт: входящие клиента переводятся тренеру на английский (оригинал под переводом);
на ОТПРАВКЕ текст из композера переводится в язык клиента (messages route уже так умеет:
`if (adminLang && clientLang && adminLang !== clientLang)` -> translateAndDetect targetLang=clientLang).

### B. Агент ВСЕГДА пишет черновик на английском
`lib/sales-agent.ts`: во всех местах, где сказано «reply in the client's language» / «client's
language (default English)» / «Indonesian clients get Bahasa» - заменить на «ВСЕГДА пиши черновик
на английском (это язык персонала студии для подтверждения); НЕ подстраивайся под язык клиента и
не зеркаль язык предыдущих сообщений в треде». Клиент всё равно получит на своём языке - перевод
делается на отправке (часть A).
- Затрагиваемые блоки промпта: SAFE/BOOKING/ESCALATE (легаси) и MEDICAL/BUSINESS (полная автономия).
- Агент читает оригинальные сообщения клиента (`m.body`) - оставить как есть (модель многоязычна,
  понимает вход на любом языке, а выход теперь фиксирован английским).

## Как это работает end-to-end после правки (Canggu)

- Клиент пишет по-английски -> агент черновик EN -> Сеня видит EN -> Send -> EN==EN, перевода нет
  -> клиент получает EN. ✓
- Клиент пишет по-русски -> входящее переведено Сене на EN -> агент черновик EN -> Сеня видит EN
  -> Send -> translateAndDetect(EN, target=ru) -> клиент получает RU. ✓
- Локал пишет на Bahasa -> Сеня видит EN -> черновик EN -> Send -> клиент получает Bahasa. ✓

## Что НЕ меняется
- Ubud и прочие студии (нет агента / другой язык персонала).
- Клиентская доставка всегда на языке клиента (уже так на отправке при заданном inboxLanguage).
- Медицинские/бизнес правила содержания (формула Андрея и т.д.) - не трогаем, меняем только ЯЗЫК
  черновика.

## Проверка после внедрения (E2E)
1. `inboxLanguage` Canggu = en (запрос к БД).
2. Живой прогон classifyForQa / реальный лид: черновик приходит на английском независимо от языка
   клиента и истории треда.
3. Смоук отправки: русскоязычному клиенту уходит русский (translatedBody=ru), англоязычному -
   английский; тренеру в обоих случаях черновик был английский.
