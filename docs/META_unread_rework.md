# МЕТАПРОМТ: переработка счётчиков непрочитанных (admin = просмотр, trainer = ответ)

Версия: 1.0 (06.07.2026). Статус: ожидает утверждения владельцем.
Решения владельца (кнопками): просмотр тренером гасит админскую цифру; ответ
админа гасит тренерскую; фильтр «Awaiting reply» добавляем.

## 1. Целевая семантика

- **unreadAdmin** (цифра у админа, на чате и в сумме): «сообщение клиента, на
  которое ещё НИКТО не взглянул и не отреагировал». Гасится ЛЮБЫМ вниманием
  сотрудника: админ открыл чат, тренер открыл чат, любой ответ, любая реакция.
- **unreadTrainer** (цифра закреплённого тренера): «сообщение клиента, на
  которое ещё никто не ОТВЕТИЛ». Гасится ТОЛЬКО ответом или реакцией
  сотрудника (любого - ответ админа закрывает вопрос, тренер не дублирует).
  Просмотр (свой или админа) НЕ гасит. Это ТЕКУЩЕЕ поведение - без изменений.
- **Бейдж на иконке приложения** = цифра роли владельца телефона: у тренера
  его unreadTrainer-сумма, у админа - unreadAdmin-сумма. Пуши несут `badge`
  с пересчитанным числом получателя; открытие инбокса и события в чате
  обновляют OS-бейдж тем же числом.
- **Фильтр «Awaiting reply»** в админском инбоксе (рядом с All/Today/Tomorrow):
  чаты, где последнее слово за клиентом (нет ответа/реакции после его
  сообщения). Без красной цифры - спокойный контрольный список, чтобы
  неотвеченные клиенты не терялись после «просмотрел и забыл».

## 2. Что реально меняется в коде (дельта)

1. **Гашение просмотром для unreadAdmin** (re-enable read-on-view, admin-only):
   при открытии чата в инбоксе (веб и WebView мобилки) любым сотрудником
   (админ ИЛИ тренер) → unreadAdmin этого чата = 0. unreadTrainer не трогаем.
   Точка: эндпоинт открытия диалога (где до 03.07 жил mark-read) или POST
   /api/whatsapp/conversations/[id]/read (восстановить/направить на admin-счётчик).
2. **unreadTrainer** - без изменений (ответ/реакция любого сотрудника гасит;
   уже так с c3b0b63/065c934).
3. **Awaiting reply фильтр**: conversations API уже возвращает сообщения/метки -
   добавить флаг awaitingReply (последнее событие в чате = INBOUND клиента без
   последующего ответа/реакции) + вкладка-фильтр в Inbox.tsx (admin only).
4. **OS-бейджи per-роль**: везде, где шлётся пуш (webhook входящих,
   booking-events) и где обновляется бейдж, считать число ПОЛУЧАТЕЛЯ:
   тренеру - сумма его unreadTrainer, админу - сумма unreadAdmin (native
   expo-push `badge` + web-push payload). Открытие инбокса ставит OS-бейдж
   в текущее число роли (46f678f уже делает - проверить, что источник = роль).
5. **Строка-пояснение в инбоксе** (добавлена 06.07): текст обновить под новые
   правила, показывать РАЗНЫЙ для админа и тренера:
   - админу: «Red numbers = messages nobody has seen yet. Opening a chat
     clears them. The Awaiting reply tab lists clients still waiting for an answer.»
   - тренеру: «Red numbers = clients waiting for a reply. They clear when
     someone replies or reacts - not when you open the chat.»
6. **Хвост**: unreadAdmin в проде сейчас = «неотвеченные» (36 у Чангу).
   Исторических событий просмотра нет, пересчитать невозможно - оставить как
   есть: админ гасит просмотром или Mark all read. Отметить Свете в отчёте.

## 3. Не трогаем

- Логику реакций/ответов (гасят оба счётчика) - уже целевая.
- Mark all read (админ) - остаётся как массовое гашение.
- Тренерский mobile-бейдж механизм (46f678f) - только источник числа сверить.

## 4. Проверка

- Сценарный прогон на scratch-БД: входящее → у обоих +1; тренер открыл →
  у админа 0, у тренера остаётся; тренер ответил → у обоих 0; админ ответил →
  у обоих 0; реакция тренера → у обоих 0; awaitingReply-флаг корректен.
- npm run build; коммит точечно (в репо чужие незакоммиченные ad-analytics
  файлы - не захватывать).
