# МЕТАПРОМТ: отмена/перенос класса тренером + пакет улучшений по инциденту 04.07

Версия: 1.0 (черновик, 05.07.2026). Статус: ожидает утверждения владельцем.
Контекст-источник: memory `project_bookgravity_session_2026-07-04.md` (форензика инцидента
Tatiana Stepanchenko / тренер Дита / класс 04.07 09:00) + код на HEAD `3b6817d`.

## 1. Цель

Тренер (или админ), который не может провести класс, за 2 нажатия отменяет или переносит
ВЕСЬ класс. Каждый записанный клиент получает тёплое человеческое сообщение с ПОНЯТНОЙ
кнопкой действия. Абонемент возвращается ровно один раз. В данных фиксируется, кто и когда
отменил. Кейс «клиент пришёл к отменённому классу» становится невозможным.

## 2. Роль исполнителя

Claude (Book_Part_5), полный доступ к репо `~/Documents/Claude/bookgravity`, прод =
Vercel (push в main), БД Turso. Тексты клиентам - только через владельца (отдельное «да»
на каждый шаблон перед подачей в Meta).

## 3. Диагноз (корень инцидента, из форензики)

1. У тренера нет инструмента «отменить/перенести класс»: только по-одной отмена брони,
   которая шлёт сухой робо-шаблон `booking_canceled` без причины и без действия.
2. `TimeSlot` не имеет статуса: отмена класса = жёсткое удаление слота (`deleteSlotCascade`),
   брони исчезают из базы бесследно (бронь Татьяны удалена полностью), клиентов никто
   не уведомляет.
3. Автор отмены нигде не фиксируется (`afterStaffCancellation` шлёт сообщение без
   fromTrainerId, у Booking нет cancelledBy) - форензика упёрлась именно в это.
4. Клиент получает «canceled» без «что дальше» - нет кнопки, растерянность.

## 4. Решение (утверждённое направление A+C + кнопки)

### 4.1 Кнопка «Can't teach this class» у тренера (и у админа)
- Тренер: в деталях слота своего расписания. Админ: в деталях слота расписания студии.
- Выбор из двух действий: **Cancel class** или **Move class** (на другой существующий
  слот или на новый, создаваемый на месте: дата + время).
- Причина - короткий выбор (sick / emergency / other), попадает в сообщение админу
  и в аудит, НЕ в сообщение клиенту (клиенту - тёплый готовый текст).

### 4.2 Cancel class (эндпоинт `POST /api/trainer/slots/[id]/cancel` + админ-аналог)
Одна транзакция + пост-эффекты:
- все CONFIRMED-брони слота → `CANCELLED` (НЕ удаление) с cancelledBy-атрибуцией;
- возврат абонементных классов ровно один раз (существующий guard: обнуление
  `membershipId` после restore);
- слот → `status: CANCELLED` (новое поле), исчезает из публичной брони и из активного
  расписания, остаётся в данных для истории/зарплат-аудита;
- каждому клиенту - тёплый шаблон **T1** с кнопкой (см. 5), вне 24ч-окна работает
  (utility-template);
- уведомление админу студии через `admin_message` («класс X отменён тренером Y,
  причина Z, клиентов N»);
- Google Calendar event удаляется/помечается (существующий googleEventId-путь).

### 4.3 Move class (эндпоинт `POST /api/trainer/slots/[id]/move`)
- Цель: существующий слот с местами ИЛИ новый слот (дата+время, тот же тренер или
  замена);
- все CONFIRMED-брони переносятся (регенерация ticket-кода - паттерн уже есть в
  reschedule), исходный слот → CANCELLED со ссылкой на новый (`movedToSlotId`);
- каждому клиенту - шаблон **T2** с двумя quick-reply кнопками (см. 5);
- вместимость цели проверяется до старта; при нехватке мест - отказ с подсказкой.

### 4.4 Замена сухого `booking_canceled` на тёплый (одиночные staff-отмены)
Тот же тон, что T1 (текст - на отдельное «да», БЫЛО → СТАНЕТ).

### 4.5 Удаление слота админом = тот же человеческий путь
`deleteSlotCascade` с CONFIRMED-бронями больше не молчит: путь «удалить класс с живыми
бронями» маршрутизируется через Cancel class (4.2: уведомления + возврат + soft-cancel).
Жёсткое удаление остаётся только для пустых слотов.

### 4.6 Атрибуция «кто отменил» (закрытие форензик-дыры)
- Booking: `cancelledAt`, `cancelledByUserId`, `cancelledByRole` (client|trainer|admin|system);
- TimeSlot: `status`, `cancelledAt`, `cancelledByUserId`, `cancelReason`, `movedToSlotId`;
- `afterStaffCancellation` и слот-отмена передают автора в `appendOutboundMessage`
  (fromTrainerId), чтобы в инбоксе было видно, от чьего имени ушло сообщение.
Все изменения схемы аддитивные (без миграции данных).

## 5. Кнопки клиенту (инфраструктура уже есть: webhook type "button" → payload)

- **T1 (отмена):** URL-кнопка **Book another day** → страница брони студии
  (самообслуживание; абонемент уже возвращён). Плюс бот понимает текстовый ответ.
- **T2 (перенос):** quick-reply **I'll be there** (payload `KEEP:<ticket>`) - бот
  подтверждает место; **Can't make the new time** (payload `DROP:<ticket>`) - бот
  отменяет бронь, возвращает абонемент, шлёт ссылку на перезапись.
- Обработчики `KEEP:`/`DROP:` добавляются в `lib/cancel-bot.ts` рядом с существующими
  `CANCEL:`/`CANCELALL:` (идемпотентно: повторный тап = повтор того же ответа).

## 7. Очерёдность (владелец выбрал: техчасть сразу)

- **Фаза 1 (сразу после «да» на план):** схема (4.6) → эндпоинты cancel/move → UI
  тренера и админа → обработчики KEEP/DROP в боте → маршрутизация deleteSlotCascade →
  локальная проверка `npm run build` + прогон сценариев на dev-БД. Отправка клиентам
  до одобрения T1/T2 идёт через существующий approved `booking_canceled` (fallback),
  чтобы фича была рабочей с первого дня; тёплые тексты подменяют его без правки кода
  (имя шаблона - конфиг).
- **Фаза 2 (после ответа Андрея/Светы + твоего «да» на тексты):** подача T1/T2 (+ тёплый
  `booking_canceled_v2`) в Meta → переключение конфигурации на новые шаблоны → сквозной
  тест на реальном номере владельца.
- **Фаза 3:** обновление хендовер-памяти + university.

## 8. Критерии качества / самопроверка

- Транзакционность: частичная отмена невозможна (или все брони, или ни одной).
- Возврат абонемента ровно 1 раз (существующий membershipId-guard переиспользован).
- Работает вне 24ч-окна (только utility-шаблоны, никакого free-form).
- TZ-aware (Бали UTC+8), изоляция студий (город - переменная, cross-studio утечек нет).
- Идемпотентность кнопок клиента (двойной тап безопасен).
- Не сломаны 7 верифицированных путей брони + существующий одиночный cancel/reschedule.
- `npm run build` перед пушем (строгий TS Vercel), деплой только в подтверждённый
  Vercel-аккаунт `hatabishchips-projects`.
- Ни одного длинного тире ни в одном человеко-видимом тексте.
