# META: мгновенный инбокс - локальный кэш + SSE-push (утв. 15.07.2026)

Только инбокс (app/_components/Inbox.tsx + новые lib/api). Серверная модель сообщений не меняется. НЕ трогать: тренер/клиент UI вне инбокса, WhatsApp-тексты, механику отправки/медиа (merge с tmp_ + localMediaCache сохранить).

## Проблема
Открытие инбокса: convos===null -> "Loading..." пока /api/whatsapp/conversations. Открытие чата: loadingDetail && !detail -> "Loading..." пока /api/whatsapp/conversations/{id}. Каждый раз ждём сеть с нуля (~1-1.5с).

## Часть A - мгновенное открытие (кэш, как WhatsApp Web)
1. lib/inbox-cache.ts: список чатов + детали в localStorage. Список - 1 ключ. Детали - LRU ~30 чатов, каждый капнут ~60 последних сообщений. Автопрунинг < ~4МБ. Версионирование ключа (bump при смене формата).
2. Гидратация convos из кэша СИНХРОННО на первом рендере (useState initializer) -> список сразу; refreshList() в фоне. "Loading" только при пустом кэше.
3. Открытие чата: detail из кэша мгновенно; refreshDetail() тихо догоняет. Спиннер только если чата нет в кэше.
4. Фоновый префетч топ-15 свежих чатов в кэш после загрузки списка (throttle, visible-tab).
5. Каждый успешный ответ пишется в кэш. Mesh с in-flight tmp_ + localMediaCache сохранить.

## Часть B - SSE-push (связь без polling)
1. app/api/whatsapp/stream/route.ts (Node runtime, ReadableStream SSE): держит соединение, шлёт event "changed" при новом/изменённом сообщении студии. Сигнал: WhatsApp-вебхук bump'ит версию-счётчик на студию (БД-поле или lightweight), стрим поллит счётчик внутри себя дёшево ИЛИ in-memory pub-sub на инстанс + версия для кросс-инстанса.
2. Клиент: EventSource -> по "changed" точечный refreshList()/refreshDetail(). Автопереподключение (EventSource делает сам).
3. Polling остаётся ТОНКИМ фолбэком (реже, ~60с) на случай обрыва/лимитов Vercel.
4. maxDuration серверлеса: стрим сам закрывается по таймеру (~50с), клиент переподключается - стандартный SSE-паттерн на Vercel.

## Этапы (каждый build -> E2E превью -> деплой)
A. Кэш + префетч (самый заметный эффект).
B. SSE + сигнал из вебхука + polling-фолбэк 60с.

## Критерии
Кэш не показывает мёртвые данные дольше 1 фонового обновления; localStorage не переполняется (прунинг проверен); пустой кэш = поведение как сейчас; ноль регрессий отправки/медиа; при обрыве SSE - polling-фолбэк держит.
