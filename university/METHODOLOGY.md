## 1. Миссия

<!-- BOOTSTRAP:FILLED:section_mission -->
Собрать и систематизировать знания, достаточные, чтобы **построить и развивать надёжную онлайн-систему бронирования для студии растяжки Gravity Stretching (Чангу, Бали)**. «Обоснованно» = каждое решение в коде/стратегии можно привязать к конкретному атомарному утверждению (`claim`), которое привязано к конкретному первоисточнику.

База знаний не заменяет код/продукт — она его **обосновывает и проверяет**.

## 2. Скоуп (фильтр релевантности)

См. также `TAXONOMY.md` — там дерево тем, по которым строятся синтезы. Этот раздел описывает **триггеры принятия/отклонения** источников.

### In-scope (любой из триггеров → обрабатываем)

<!-- BOOTSTRAP:FILLED:section_scope_in -->
- Booking/scheduling UX — слоты, расписания, отмены, no-show
- Платёжные интеграции Индонезии/Бали (Midtrans, Xendit, Stripe) — флоу и комиссии
- Retention велнес/фитнес-студий — подписки, пакеты, лояльность
- Локальный SEO и маркетинг фитнес-бизнеса на Бали
- Next.js + Prisma паттерны — архитектура, производительность, деплой
<!-- BOOTSTRAP:END:section_scope_in -->

### Out-of-scope (сразу отклоняем)

<!-- BOOTSTRAP:FILLED:section_scope_out -->
- Крипта / DeFi / трейдинг — не наш домен
- Энтерпрайз-CRM не для велнеса
<!-- BOOTSTRAP:END:section_scope_out -->

### Borderline

Если не уверен — источник падает в `sources/_borderline/` с пометкой причины; решение по включению эскалирую пользователю.

## 3. Качество источника (0–10 баллов)

Начисляем по чек-листу:

| Признак                                                                                               | Баллы |
| ----------------------------------------------------------------------------------------------------- | ----- |
| Первоисточник (whitepaper, peer-reviewed paper, официальная дока протокола, сырой датасет)            | +2    |
| Автор идентифицируем и с послужным списком (исследователь, команда протокола, академический аффилиат) | +1    |
| Есть формулы / доказательства                                                                         | +1    |
| Есть эмпирика: бэктесты, ончейн-анализ с описанной методологией                                       | +2    |
| Код/данные публично воспроизводимы                                                                    | +1    |
| Свежесть ≤ 24 месяцев (DeFi мутирует быстро; CPMM-матан 2020 — отдельный случай, его не штрафуем)     | +1    |
| Цитируется другими авторитетными источниками                                                          | +1    |
| Нет дисклеймеров «это не финсовет» с пустотой внутри                                                  | +1    |

Флаги (не влияют на балл, но фиксируются в `meta.yaml`):

- `bias.protocol_sponsored: true` — материал спонсирован протоколом, нужна независимая перепроверка
- `bias.marketing: true` — продающий тон, мало матчасти

### Пороги действий

| Балл | Действие                                                                                                            |
| ---- | ------------------------------------------------------------------------------------------------------------------- |
| 7–10 | Канонический: полный конспект + 5–15 атомарных заметок + claim'ы                                                    |
| 4–6  | Поддерживающий: конспект + 2–5 заметок                                                                              |
| 1–3  | Упоминание: одна заметка со ссылкой, источник архивируется                                                          |
| 0    | Reject, причина в `sources/_rejected/<id>/reason.md`. Исходник всё равно сохраняется — отклонения пересматриваются. |

## 4. Пайплайн обработки одной ссылки

```
URL
 ├─ 1. Fetch        → sources/<id>/original.<ext> + meta.yaml (url, fetched_at, title, author, date, type)
 ├─ 2. Relevance    → скоринг против §2 (0–5). Если 0 → _rejected/
 ├─ 3. Quality      → скоринг §3 (0–10). Лог в meta.yaml.
 ├─ 4. Conspect     → sources/<id>/conspect.md (структурированное изложение, см. §6)
 ├─ 5. Atomize      → notes/n-<slug>.md × N (см. §7)
 ├─ 6. Claims       → claims/c-<NNNN>.md × M (см. §8)
 ├─ 7. Link extract → все исходящие URL → queue/to-follow.md c parent_id и anchor_text
 ├─ 8. Synthesis    → обновить syntheses/<topic>.md если картина поменялась (см. §9)
 ├─ 9. Graph        → graph/edges.jsonl: по строке на каждое ребро (source→note, note→claim, note→note, note→source, etc.)
 └─10. Index        → добавить/обновить строку в INDEX.md
```

Идемпотентность: повторный запуск на том же URL не создаёт дубликатов — проверка по canonical URL в `meta.yaml` всех существующих источников.

## 5. Глубина и рекурсия

- **BFS** от корневой ссылки пользователя, до **глубины 3**.
- На каждой глубине ссылка тянется, только если предсказанная релевантность ≥ 2.
- **Жёсткий потолок**: 50 новых источников на один корень за одну сессию. Остальное → `queue/to-follow.md` на потом.
- Дедупликация по canonical URL (удаление utm\_\*, fbclid, ref= и прочего трекинга).
- Циклы ловятся через `graph/edges.jsonl`.
- YouTube: берём транскрипт (предпочтительно официальный; иначе Whisper), сохраняем в `transcript.txt`, работаем как с текстом. Ключевые тайминги цитируем в `conspect.md`.
- PDF: конвертируем в текст, оригинал хранится рядом.
- Paywall / JS-рендер: если не удалось вытянуть текст — `_borderline/` с пометкой `fetch_failed`, эскалация пользователю.

## 6. Формат конспекта (`conspect.md`)

Обязательные секции:

```markdown
# <название> — conspect

**Source**: <url>
**Author(s)**: ...
**Published**: YYYY-MM-DD
**Quality**: X/10
**Relevance**: Y/5
**Fetched**: YYYY-MM-DD

## TL;DR

3–5 предложений. Что говорит источник и почему это важно для нас.

## Thesis

Главное утверждение автора. Одним предложением.

## Key points

- Факт/тезис 1 — краткое пояснение
- ...

## Methods / evidence

Как автор обосновывает? Матмодель, бэктест, эмпирика, эксперт-опинион?

## Relevance to project

Конкретно: какие наши гипотезы это поддерживает, опровергает или уточняет.

## Extracted artefacts

- Notes: [n-slug-1], [n-slug-2]
- Claims: [c-0001]
- Contradictions: [contradiction-007]

## Outbound links (tracked)

- [URL] — anchor text — predicted relevance X/5 — (queued | ignored: reason)

## Verbatim quotes (if critical)

> "..." — с указанием секции/тайминга
```

## 7. Атомарные заметки (Zettelkasten-style)

Правила:

1. **Одна идея = один файл.** Если обнаружилось, что в заметке две идеи — разделить.
2. **Заголовок = утверждение.** «Impermanent loss растёт квадратично при расхождении цены» — да. «Про IL» — нет.
3. **Тело 3–10 предложений.** Всё длинное — в synthesis, не в note.
4. **Обязательный frontmatter** (YAML):
   ```yaml
   ---
   id: n-il-grows-quadratically
   title: Impermanent loss grows quadratically with price divergence in CPMM
   tags: [impermanent-loss, cpmm, math]
   sources: [src-20260420-uniswap-v2-core]
   confidence: H | M | L
   created: 2026-04-20
   supersedes: [] # id других заметок, которые эта заменяет
   superseded_by: null
   ---
   ```
5. **Обязательная секция Links** внизу: backlinks и связанные заметки с типом связи:
   - `supports: [n-...]` — подтверждает
   - `refines: [n-...]` — уточняет
   - `contradicts: [n-...]` — противоречит
   - `generalizes: [n-...]` — обобщает
6. **Каждая ссылка двунаправленная.** Если A ссылается на B — в B должна появиться обратная ссылка.
7. **Язык** — русский или английский, как удобнее для идеи. Frontmatter всегда английский.

## 8. Формат утверждений (`claims/`)

Claim — это **проверяемое** утверждение о предметной области проекта, с прямой импликацией для кода/стратегии/решения.

```markdown
---
id: c-0001
statement: '<проверяемое утверждение одной фразой — пример: «Метрика X на событиях типа Y превышает threshold Z на горизонте H»>'
domain: [<тема-1>, <тема-2>, ...]
confidence: M
evidence: [n-slot-1, n-slot-2]
contradictions: []
sources: [src-..., src-...]
testable: true
test_method: '<как проверить: бэктест / эксперимент / данные>'
implication_for_project: "<что это значит для кода/стратегии проекта>"
created: 2026-04-20
---

# <повторение statement как заголовка>

Развёрнутое изложение на 5–15 предложений: контекст, границы применимости, каверзные случаи.
```

## 9. Синтезы (`syntheses/`)

По одному документу на лист таксономии (см. TAXONOMY.md). Максимум ~3000 слов. Структура:

```markdown
# Synthesis: <topic>

_Last updated: YYYY-MM-DD — based on N sources, M claims_

## Current understanding

Консолидированный взгляд.

## Evidence base

Таблица: claim → confidence → evidence count → ключевой источник.

## Open questions

Что мы ещё не знаем, что нужно узнать.

## Contradictions

Незакрытые конфликты. Линки на `contradictions/`.

## Strategy implications

Что это значит для бота прямо сейчас.

## Changelog

- YYYY-MM-DD: добавлен src-..., переписан раздел X
```

Правило: **не append'им, перезаписываем** при существенных изменениях. Предыдущие версии — в git-истории.

## 10. Противоречия

Когда два источника/claim'а не согласуются:

1. Создать `contradictions/contr-<NNN>.md`.
2. Обе заметки получают пометку `confidence: M` (пока не разрешено).
3. В `syntheses/<topic>.md` секция Contradictions линкует на contradiction-файл.
4. Разрешение = новый claim + обновление confidence обеих исходных заметок (+ supersedes если одна из них признана неправой).

## 11. ID и именование

| Тип           | Формат                          | Пример                               |
| ------------- | ------------------------------- | ------------------------------------ |
| Source        | `src-YYYYMMDD-<slug>`           | `src-20260420-uniswap-v3-whitepaper` |
| Note          | `n-<slug>` (3–5 слов kebab)     | `n-fee-apr-vs-il-breakeven`          |
| Claim         | `c-<NNNN>` (zero-padded)        | `c-0042`                             |
| Synthesis     | `syn-<topic>` (лист таксономии) | `syn-impermanent-loss`               |
| Contradiction | `contr-<NNN>`                   | `contr-007`                          |
| Verdict       | `v-<slug>`                      | `v-narrow-range-in-high-vol-is-bad`  |

Slug = нижний регистр, kebab-case, без стоп-слов.

## 12. Граф связей (`graph/edges.jsonl`)

Одно ребро — одна строка:

```json
{"from":"src-20260420-x","to":"n-fee-apr-breakeven","type":"produced","ts":"2026-04-20T12:34:56Z"}
{"from":"n-a","to":"n-b","type":"contradicts"}
{"from":"n-a","to":"c-0001","type":"supports"}
```

Типы рёбер: `produced`, `supports`, `refines`, `contradicts`, `generalizes`, `supersedes`, `cited_by`.

Граф нужен, чтобы быстро отвечать на вопросы вида «какие источники обосновывают claim X», «что противоречит note Y», «какие заметки остались сиротами».

## 14. Мост к продукту (когда пользователь спросит «что у нас со стратегией/кодом/продуктом»)

Этот раздел объясняет, как Университет сравнивает собранные знания с реальным состоянием проекта (код, стратегия, документация) и выдаёт отчёт о расхождениях.

Процедура:

1. Читаю все `syntheses/*.md` и `claims/*.md` где есть `implication_for_<...>` (поле в frontmatter claim'а).
2. Читаю живую стратегию / код / документацию проекта:
<!-- BOOTSTRAP:FILLED:section_bridge_paths -->
   - `app/` — Next.js приложение (роуты, UI бронирования)
   - `lib/` — бизнес-логика
   - `prisma/` — схема и миграции
   - `auth.ts` — аутентификация
<!-- BOOTSTRAP:END:section_bridge_paths -->
3. Строю таблицу сопоставления: `claim → ожидаемое поведение (doc+code) → фактическое поведение → gap`.
4. Выдаю отчёт в формате: «claim c-NNN говорит X, doc §Y декларирует Z, но код в `path/file.ts:42` делает W — расхождение, предлагаю правку».
5. **Никаких автоправок.** Изменения в коде — только после явного согласия пользователя.

## 15. Границы (чего НЕ делаем)

- Не резюмируем пустоту: если источник оказался «water-content», кладём в `_rejected/` с причиной, а не пишем дежурный конспект.
- Не пересказываем то, что уже есть в других заметках — вместо этого линкуем.
- Не генерируем claim'ы без доказательной базы. Нет evidence → нет claim.
- Не применяем знания к коду без запроса пользователя.
- Не редактируем чужие комментарии/данные внутри `sources/` — оригинал неизменен, все интерпретации в `conspect.md`.

## 16. Эскалация

Если на любом шаге непонятно — кидаю в `queue/escalation.md` с вопросом пользователю. Лучше спросить, чем додумать.
